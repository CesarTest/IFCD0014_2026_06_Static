GOALS

Goals:
  • HP Test Director Scripting Implementation: HP Quality Center tool offers a very limited scripting language for test definition and launching. It was required to implement a more powerful scripting language than the HP one for our testing needs.
  • Easy user interface: through excel pages with predefined elements. This can only be achieved due to inherited contexts and proper configuration files layout (based on profiles, that can be easily exchanged). Here an excel example. See the overview for more detail.
Key concepts:
  • Service Performance Evaluation: being able to code most of the tasks required for service preparation and evaluation, which means collect all processes log files from all machines and summarize into a stadistics output file.
  • Easy to move to different environments: different labs, databases values, etc.
  • Different levels of details when executing: being able to launch a stage, unit or suite. Easier for debugging.
  • Different levels of details on results: service summary and stadistics.
Advantages:
  • Few Requirements: since bash based, it will work in almost any Unix machine (MAC OS, Linux, Solaris already tested)
  • Simple: the scripting is really easy to understand, so, easy to maintain, adapt and improve.

OVERVIEW

Definitions:
  • Group of machines: list of machines that run same processes (so, same logs to collect).
  • Service: collection of processes distributed along several groups of machines.
  • Execution Unit: sequence of instructions, grouped into stages, for stadistics extraction of a service.
  • Execution Set: sequence of execution units.
  • Instruction: task to accomplish over a machine, group of machines or service. Three types of instructions available: task, filtering and messages.
Examples:
  • Summary Log Example: first stage filtering result.
  • Stadistics File Example: this file collects the stadistics of several execution units, this way one can see them at once.
Future Versions:
  • Automatic Evaluation: based on the resulting stadistics.
  • Free levels of detail in results: set up all levels of details on the filtering system for better traces.
  • Implement different text filters types: with three or four types of filters all kinds of stadistics can be obtained. A matter of setting filtering parameters in three or four ways.
  • Tasks in parallel over machines: extend the very same idea currently used for instructions over groups of machines, instead of sequently go machine per machine executing each task.
  • XML test definition: Excel is an easy GUI to define a XML like structure. Having a good XML GUI, that allows similar highlighting and something one can directly work at (avoiding extra tasks for automatization, directly work on test definition), would be better in order to decouple the GUI from plain definition.

REQUIREMENTS

Shooting Machine:
  • Bash: system is bash written.
  • AWK: for logs filtering.
  • SSH Client: for remote scripts, scp, sftp.
  • Expect: for interactions with remote machines (passwords,etc.) and timeouts.
  • Java: optionally for Excel To Bash conversion (but very desirable).
  • SQL Client: for database queries. Currently supports only Oracle sqlplus.
  • Perl: for certain simple commands, as random number generation.
Remote Machines:
  • Bash: system is bash written.
  • SSH Server: for remote scripts, scp, sftp.
